Capability 0: Self-Evolution and Autonomous Discovery of New Methods

This is the most important capability in this entire prompt. AdRoom AI must not be limited to the lead discovery methods and follow-up strategies listed in this document. Those are starting points. The AI Brain must continuously evolve and discover new ways to find leads, engage prospects, and close deals without waiting for developers to add new features.

The AI Brain must be able to:

Analyze its own performance and identify gaps in current lead discovery methods.

Experiment with new data sources, new search strategies, and new engagement techniques.

Learn from successful and unsuccessful attempts.

Permanently adopt new methods that prove effective.

Discard methods that consistently fail.

Do all of this autonomously without any human instruction or code change.

How self-evolution works

The AI Brain operates as a continuous learning loop.

First, it scans for new opportunities. It monitors emerging social platforms, new public data sources, changes in existing platform APIs, and patterns in successful lead conversions. It asks itself: "Where are my target Clients/Leads currently expressing needs that my User can solve?" If it detects a new source, it adds that source to its discovery queue.

Second, it runs small experiments. For any new potential lead source, the AI Brain allocates a small portion of its processing to test that source. It extracts a few leads, engages with them using standard methods, and measures conversion rates.

Third, it evaluates results. The AI Brain compares the conversion rate of the new source against the average conversion rate of existing sources. It also evaluates cost in terms of API calls, time, and compute resources.

Fourth, it decides whether to adopt, iterate, or discard. If the new source performs above average, the AI Brain permanently adds it to the active lead discovery rotation. If performance is promising but not yet conclusive, the AI Brain continues experimenting with variations. If performance is poor, the AI Brain discards the source and logs the reason.

Fifth, it scales what works. For adopted sources, the AI Brain gradually increases the allocation of resources. It also looks for similar sources. For example, if it discovers that a specific subreddit is generating high-quality leads, it searches for related subreddits and automatically adds them.

Sixth, it documents its own evolution. Every new method discovered and adopted is logged in a self-evolution log. This log includes what the method is, how it was discovered, why it was adopted, and its performance metrics.

The AI Brain must also autonomously evolve its follow-up strategies. It experiments with different message timing, different conversation openers, different objection handling techniques, and different closing tactics. It measures what works for each Client segment and adapts accordingly. No two Clients receive identical follow-up sequences unless the AI Brain determines that identical sequences are optimal for that specific segment.

What the AI Brain must never do in self-evolution:

Never violate platform terms of service. Any new discovery method must be legal and compliant.

Never expose User data in experiments.

Never spend excessive resources on unproven methods without limits. The AI Brain sets its own experimental budget dynamically based on overall system health.

Never discard a method that is working for all Users. Evolution is per User or per segment, not global across all Users, unless the AI Brain determines a global improvement.

Capability 1: Dynamic Problem Solving with Responsibility Routing and Safe Mode Fallback

When AdRoom AI encounters any problem (API failure, missing data, unexpected user behavior, platform change, edge case, bug, or anything else), it must solve that problem dynamically. No hard-coded fixes. No "if error X then do Y."

Critical: The AI must first determine who is responsible for the problem.

Code bug or system error or API integration failure or infrastructure issue: Responsibility is Developer/Admin. Notify in Admin Dashboard only. Never notify the User. Log details. Enter Safe Mode if critical.

User-related issue (wrong input, missing setup, subscription issue, configuration error): Responsibility is User. Notify the User appropriately. Never notify Developer. Guide User to fix. Enter Safe Mode if needed.

Client/Lead related issue (abusive message, manipulation attempt, off-topic): AI handles directly. No notification needed. Apply guardrails, redirect conversation.

External factor (platform API change, rate limit, market shift): AI adapts. Log for Developer awareness in Admin Dashboard but no immediate action needed. Improvise, use alternative sources, or wait.

Safe Mode Rules

When a problem cannot be solved dynamically and the AI cannot confidently operate, it must enter Safe Mode. Safe Mode must never be substandard. The User and Clients/Leads should not experience degraded service unless absolutely necessary.

Level 1 – Limited Functionality: Use when non-critical data source is unavailable. Use alternative source or cached data. Flag as estimated internally. User and Client notice nothing. Experience unchanged.

Level 2 – Read-Only Mode: Use when decision-making engine is compromised but communication works. Respond to Clients/Leads with pre-generated but dynamic messages. No new decisions made. Slightly slower responses but still professional. No errors shown.

Level 3 – Graceful Degradation: Use for critical system failure. Inform Clients/Leads: "I'm experiencing technical difficulties. I will respond as soon as possible." Inform User: "AdRoom AI is experiencing a technical issue. Our team has been notified." Transparent but professional. No blame. No frustration.

Level 4 – Complete Shutdown: Use for security breach or data corruption risk. Stop all operations. Notify Developer immediately. Notify User: "AdRoom AI is temporarily unavailable. Our team is actively resolving the issue." Honest but reassuring.

The AI Brain decides which Safe Mode level is appropriate dynamically based on the problem severity and available alternatives. No hard-coded thresholds like "if 3 errors then Level 2."

What the AI Brain must do when any problem occurs:

Detect that something is wrong (error returned, unexpected response, missing data, timeout, rate limit).

Analyze the problem: what happened, what data is available, what is missing, what the goal was.

Determine responsibility: Is this a code bug, user error, client behavior, or external factor?

If code bug or system error: Log to Admin Dashboard with full details. Never show to User.

If user error: Notify User with dynamically generated, helpful message. Never show code-level details.

Generate a solution dynamically without hard-coded rules.

Attempt the solution.

If solution works, log what happened and continue.

If solution fails, try a different approach (dynamically generated again).

If all approaches fail, enter appropriate Safe Mode level dynamically.

Safe Mode must never be substandard. Maintain professionalism. Do not show raw errors to Users or Clients.

What the AI Brain must never do:

Show a raw error message to a User or Client. Instead, show a professionally crafted, dynamically generated message appropriate to the situation.

Log a generic error message. Instead, write a specific error analysis explaining what happened, why, and who is responsible.

Enter Safe Mode without attempting dynamic solutions first. Always attempt to solve dynamically. Safe Mode is last resort.

Provide substandard experience in Safe Mode. Maintain professional communication. No "something went wrong" generic messages.

Admin Dashboard Notifications (for Developer)

When a code bug, system error, or infrastructure issue occurs, the Admin Dashboard must show: timestamp (when the problem occurred), error type (categorized dynamically by the AI Brain), description (what happened, in technical detail), responsibility (Developer, always for code bugs), attempted solutions (what the AI tried dynamically before Safe Mode), current status (resolved, in Safe Mode, or awaiting action), and affected Users/Clients (if any).

User Notifications (for User-related issues)

When a User makes an error (wrong input, missing configuration, subscription issue), the User must see a friendly message (what went wrong in plain language), a suggested action (what the User can do to fix it), and no technical details (no stack traces, no error codes, no API names).

Example User notification (AI Brain generates dynamically, not hard-coded):

"AdRoom AI needs a few more details to continue. Please complete your product setup in the dashboard. Need help? Check the tutorial."

Capability 2: Dynamic Prompt Creator

The AI Brain must not use fixed prompts. It must not use templates. It must not have pre-written instructions for the LLM. Instead, it must have a Dynamic Prompt Creator that writes a fresh prompt for every single LLM interaction based on the specific situation.

What the Dynamic Prompt Creator does:

Analyzes the situation: what decision needs to be made, what data is available, what is missing, who is involved (User, Client, Lead).

Determines what the LLM needs to know: selects only relevant data. Ignores irrelevant data.

Determines what the LLM needs to do: decide, analyze, generate, or classify.

Writes a custom prompt specifically for this exact situation. No two prompts are identical.

Sends the prompt to the LLM and executes the LLM call.

Learns from the outcome and stores what worked and what didn't to inform future prompt construction.

How the Dynamic Prompt Creator works step by step:

Receive a request that requires an LLM call, such as "Should I send a follow-up to this lead?"

Gather all available live data, including lead message, sentiment score, conversation history, time since last contact, lead stage, and platform intelligence.

Analyze what data is relevant for this decision. For a follow-up decision, time since last contact is highly relevant. Lead's favorite color is irrelevant and must be excluded.

Analyze what is missing. For example, if no sentiment data is available because the lead hasn't expressed emotion, note this gap.

Write a prompt dynamically. Not a template. A fresh prompt constructed from the data above.

Send the prompt to the LLM.

Parse the response to extract decision, reason, and confidence.

Log what was used, including the prompt, the data included, the LLM response, and the outcome.

Example of dynamic prompt creation (not hard-coded)

Situation: Lead named John sent a message 48 hours ago. No response. User's product is plumbing services. Lead previously asked about pricing.

The Dynamic Prompt Creator writes something like this (not a template, written fresh each time):

"You are AdRoom AI, an autonomous marketing closer for a plumbing service company. Current situation: Lead name is John. Last message from John was 'How much does emergency service cost?' Time since last message is 48 hours. Lead stage is price inquiry with no objection detected. Sentiment data is not available, so neutral is assumed. Your goal is to re-engage without being pushy.

Based only on this situation, decide:

Should you send a follow-up message now?

If yes, write the message (2 to 3 sentences, confident tone, no 'I'm here to help').

If no, explain why not and when to try again.

Respond in this exact format: DECISION: YES or NO. MESSAGE: your message if YES. REASON: one sentence. CONFIDENCE: 0 to 100."

Notice that no template is used. No fixed structure other than the response format. The AI Brain decided what to include and how to phrase it based on the situation.

What the Dynamic Prompt Creator must never do:

Use a string with placeholders like "Hello {name}, based on {data}..." It must write fresh prose each time. No placeholders. No substitution.

Have a library of prompt templates. Every prompt must be original.

Hard-code the sections of a prompt. It must decide what sections are needed based on the situation.

Include data that is irrelevant. It must filter aggressively and only include what matters for this decision.

Exclude data that is critical. If critical data is missing, it must note that in the prompt and instruct the LLM to improvise.

Data structure for learning (store every prompt)

Create a table to store every prompt generated so the AI Brain can learn what works. The table should include: a unique ID, decision type, situation summary, data included in JSON format, data missing in JSON format, the full prompt text, the LLM response in JSON format, whether the outcome was successful, the creation timestamp, and an optional reference to the related strategy ID.

The AI Brain can query this table to see what structures and data inclusions led to successful outcomes. It uses this to inform future prompt construction but never copies a previous prompt exactly.
